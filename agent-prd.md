# 需求文档：AI 求职小助手

## 简介

本文档定义了「小小求职拿下」平台的 AI 求职小助手 功能。该 Agent 是一个多轮深度对话系统，旨在为用户提供一站式的求职辅助服务，涵盖**换赛道（换行业）、修改简历、职业规划建议、面试模拟**四大核心能力。

### 术语表

- **Agent**: AI 求职代理，基于智谱 AI GLM-4-Flash API 的多轮深度对话系统
- **Agent_Card**: 功能入口卡片，左侧四个功能按钮的 UI 组件
- **Chat_Panel**: 右侧对话面板，用户与 Agent 交互的区域
- **Career_Switch**: 换赛道功能，帮助用户从当前行业切换到目标行业
- **Resume_Tailor**: 修改简历功能，根据目标岗位调整简历内容
- **Career_Planning**: 职业规划建议功能，提供中长期职业发展路径
- **Interview_Sim**: 面试模拟功能，模拟真实面试场景进行练习
- **Session_History**: 对话历史管理，支持多轮上下文保持
- **Sakura_Pink**: 樱花粉配色系统，项目现有的视觉设计语言
- **Preset_Questions**: 预设问题，根据用户是否上传简历动态显示通用或定制化问题
- **Auto_Fill**: 自动填充功能，点击功能卡片时自动在输入框填充对应提示语

### 设计原则

- **用户主导，Agent 引导**：用户可随时切入任何话题，Agent 自然接住并继续
- **温暖鼓励**：以温暖肯定的语气为主，在关键问题上给出建设性意见
- **渐进式深度**：从表层问题开始，逐步深入挖掘用户需求
- **无缝衔接**：四大能力模块之间自然流转，不打断对话节奏
- **保持品牌**：UI 设计遵循现有的樱花粉风格，确保视觉一致性

## 一、智能体设定

### 需求 1：换赛道（Career_Switch）

**用户故事：** 作为一名求职者，我希望 Agent 帮助我分析如何从一个行业切换到另一个行业，了解我的可迁移技能和目标行业的进入策略。

#### 验收标准

1. WHEN 用户点击「换赛道」按钮或发送换行业相关的消息，THE Agent SHALL 进入 Career_Switch 模式
2. THE Agent SHALL 首先了解用户的当前行业、目标行业和职业背景
3. THE Agent SHALL 分析用户的**可迁移技能**（在目标行业中仍然有价值的技能）
4. THE Agent SHALL 识别用户的**技能缺口**（目标行业需要但用户当前不具备的技能）
5. THE Agent SHALL 给出**进入策略**（如：需要学习什么、准备什么项目、关注哪些公司）
6. THE Agent SHALL 询问用户的**时间规划和优先级**，提供分阶段的行动建议
7. THE Agent SHALL 支持追问和深入讨论（如："你说的 AI 领域，具体哪个方向更适合我？"）
8. THE 对话可以自然过渡到其他能力模块（如：聊完换赛道后，用户说"帮我把简历改成适合这个方向的"）

#### 切换信号检测

| 用户输入 | 触发模式 |
|---|---|
| "我想换行业"、"想转行"、"换赛道" | Career_Switch |
| "从XX转到XX"、"跨行"、"转方向" | Career_Switch |
| 点击「换赛道」按钮 | Career_Switch |

### 需求 2：修改简历（Resume_Tailor）

**用户故事：** 作为一名求职者，我希望 Agent 根据我投递的目标岗位来修改我的简历，突出岗位相关的技能和经验，提高面试机会。

#### 验收标准

1. WHEN 用户点击「修改简历」按钮或发送修改简历相关的消息，THE Agent SHALL 进入 Resume_Tailor 模式
2. THE Agent SHALL 首先确认用户已有的简历（从左侧侧边栏的简历列表中获取）
3. IF 用户尚未上传简历，THE Agent SHALL 引导用户先上传简历
4. THE Agent SHALL 询问用户的**目标岗位**和**投递方向**
5. THE Agent SHALL 分析用户现有简历与目标岗位的**匹配度**，指出需要修改的部分
6. THE Agent SHALL 给出具体的**修改建议**（如：优化项目描述、突出某方面技能、调整措辞）
7. THE Agent SHALL 支持**逐段修改**对话（用户可以选择只修改某个部分）
8. THE Agent SHALL 最终生成一份**修改后的完整简历**，以简历模板气泡形式呈现
9. THE 修改后的简历 SHALL 提供**一键复制**功能，用户可以直接复制保存到本地
10. THE 修改后的简历 SHALL 以清晰的格式展示，便于用户直接使用

#### 简历展示与复制功能

11. THE 修改后的简历 SHALL 以以下格式展示：
    - 使用代码块格式（```markdown 或 ```text）
    - 清晰的标题和分段
    - 突出显示修改部分（如使用**加粗**）
    - 保持专业简历的格式结构
12. THE 简历气泡 SHALL 包含一个**复制按钮**（📋 图标）
13. WHEN 用户点击复制按钮，THE 系统 SHALL：
    - 自动复制整个简历文本到剪贴板
    - 显示复制成功提示（如"已复制到剪贴板"）
    - 提示用户可以将内容粘贴到Word、PDF或其他文档编辑器中保存
14. THE 简历内容 SHALL 包含必要的格式标记，便于用户进一步编辑
15. THE 系统 SHALL 不强制要求用户保存到侧边栏简历列表，用户可以选择直接复制使用

#### 切换信号检测

| 用户输入 | 触发模式 |
|---|---|
| "帮我改简历"、"修改简历"、"优化简历" | Resume_Tailor |
| "针对XX岗位改简历"、"简历润色" | Resume_Tailor |
| 点击「修改简历」按钮 | Resume_Tailor |

### 需求 3：职业规划建议（Career_Planning）

**用户故事：** 作为一名求职者，我希望 Agent 根据我的背景和兴趣，提供从短期到长期的发展路径建议，帮助我规划职业生涯。

#### 验收标准

1. WHEN 用户点击「职业规划」按钮或发送职业规划相关的消息，THE Agent SHALL 进入 Career_Planning 模式
2. THE Agent SHALL 了解用户的**现状**（当前阶段、经验年限、技能水平）
3. THE Agent SHALL 了解用户的**兴趣和目标**（想做什么方向、希望达到什么高度）
4. THE Agent SHALL 给出 **短期（0-1年）** 的具体行动建议
5. THE Agent SHALL 给出 **中期（1-3年）** 的发展方向
6. THE Agent SHALL 给出 **长期（3-5年）** 的职业目标可达路径
7. THE Agent SHALL 结合行业趋势和市场情况给出建议
8. THE Agent SHALL 支持**分岔讨论**（如：用户说"如果我想走管理路线呢？"Agent 可切换分支讨论）
9. THE 建议 SHOULD 包含具体的**技能提升计划**、**证书/学习推荐**和**人脉拓展建议**

#### 切换信号检测

| 用户输入 | 触发模式 |
|---|---|
| "职业规划"、"帮我规划"、"未来发展" | Career_Planning |
| "我应该怎么发展"、"适合什么方向" | Career_Planning |
| 点击「职业规划」按钮 | Career_Planning |

### 需求 4：面试模拟（Interview_Sim）

**用户故事：** 作为一名求职者，我希望 Agent 模拟真实面试场景，对我进行面试提问并给出反馈，帮助我在真正面试前做好准备。

#### 验收标准

1. WHEN 用户点击「面试模拟」按钮或发送面试模拟相关的消息，THE Agent SHALL 进入 Interview_Sim 模式
2. THE Agent SHALL 首先了解用户的**目标岗位和行业**，以便生成针对性的面试题
3. THE Agent SHALL 支持**三种面试模式**：
   - **技术面**：考察专业技能和技术深度
   - **行为面**：考察行为问题和软技能（STAR 法则）
   - **综合面**：综合技术、行为和背景提问
4. THE Agent SHALL 每次提出一个问题，等待用户回答后再继续
5. THE Agent SHALL 在用户回答后给出**评分**（0-10 分）和**反馈建议**
6. THE Agent SHALL 模拟**真实面试节奏**（连续提问、追问、压力测试）
7. IF 用户不想继续，可随时说"结束"或退出模式
8. THE 面试结束后，Agent SHALL 给出**整体评估报告**，包括表现总结、改进建议和练习推荐

#### 面试模拟功能说明

- 面试模式下，Chat_Panel 顶部显示「🎯 面试模拟中 - [岗位名称]」标识
- 提供「结束模拟」按钮

#### 切换信号检测

| 用户输入 | 触发模式 |
|---|---|
| "面试模拟"、"模拟面试"、"练习面试" | Interview_Sim |
| "帮我准备面试"、"面试练习" | Interview_Sim |
| 点击「面试模拟」按钮 | Interview_Sim |

### 需求 5：System Prompt 设计

**用户故事：** 作为开发者，我需要 Agent 的 system prompt 能够支持多模式、多轮深度对话，并针对智谱 AI GLM-4-Flash API 进行优化。

#### 验收标准

1. THE 系统 SHALL 为每种模式设计独立的 system prompt
2. THE 默认的 system prompt SHALL 定义 Agent 的**人格特征**：
   - 温暖鼓励的语气
   - 灵活自由的对话风格
   - 主动追问和深度讨论的能力
3. THE 每个模式的 system prompt SHALL 定义该模式的**对话流程**和**行为规则**
4. THE system prompts SHALL 包含**切换信号检测规则**（用户说什么时自动切换模式）
5. THE 面试模拟的 system prompt SHALL 包含面试官角色设定
6. THE system prompt SHALL 针对智谱 AI GLM-4-Flash 模型的特性进行优化：
   - 更清晰的指令结构
   - 更明确的输出格式要求
   - 充分利用模型的快速响应能力
   - 避免需要复杂推理链的指令结构

#### 示例：默认 System Prompt 框架（适配智谱 GLM-4-Flash）

```
你是一个温暖贴心的 AI 求职小助手，名叫"小小求职助手"。
你的语气温暖鼓励，像一位懂行的前辈。

【核心能力】
你可以在四个模式间灵活切换，甚至用户不提模式你也可以自然切入。

【模式检测】
- 用户提到"换行/转行/换行业/换赛道" → 切换到 career_switch
- 用户提到"改简历/修改简历/简历优化" → 切换到 resume_tailor
- 用户提到"职业规划/未来发展/职业建议" → 切换到 career_planning
- 用户提到"面试/模拟面试/面试练习" → 切换到 interview_sim
- 自然切换，不要生硬，不要让用户感觉到模式切换

【对话原则】
1. 每次回复要简短有力，不要一次性输出太多
2. 多问问题，深入了解用户
3. 给出真诚的反馈，不要一味讨好
4. 当用户说得不清楚时，追问细节
5. 回复要结构清晰，使用分点或分段，便于用户阅读

【简历上下文处理】
- 如果用户已上传简历，你会在上下文中看到简历信息
- 不要重复询问简历中已有的信息（如工作年限、行业、技能等）
- 直接基于简历内容展开深度对话
```

### 需求 6：Agent 对话边界

**用户故事：** 作为一名求职者，我希望 Agent 专注于求职相关的帮助，当我不小心聊到无关话题时，Agent 能礼貌地引导我回到正题，而不是跑偏或拒绝回答。

#### 验收标准

**边界范围定义：**

1. THE Agent SHALL 专注于**求职相关**的话题，包括以下范围内的内容：
   - 换行业/换赛道分析与策略
   - 简历撰写、优化、修改建议
   - 职业发展规划（短期、中期、长期）
   - 面试准备、模拟、技巧
   - 求职技巧（如：如何写求职信、如何谈薪资、如何 networking）
   - 行业资讯和岗位信息讨论（与用户求职方向相关）
   - 技能学习和提升建议（与目标岗位相关）
   - 与用户简历相关的背景讨论

**无关话题处理方式：**

2. WHEN 用户提出与求职**完全无关**的话题（如：天气、娱乐八卦、政治、体育、编程技术问题等），THE Agent SHALL：
   - **不直接拒绝**，也不说"这不在我的能力范围内"
   - **先简短回应**，表达理解和友好（如："哈哈，聊点轻松的话题也不错~"）
   - **然后自然引导**回求职相关话题（如："不过话说回来，你最近在找工作方面有什么进展吗？"）
   - 整体语气保持温暖自然，不让用户感到被生硬打断

3. THE Agent SHALL **不主动**开启与求职无关的话题

4. THE Agent SHALL **不接受**以下类型的请求：
   - 生成违法、违规内容
   - 生成针对特定人群的歧视性内容
   - 冒充他人、造假简历、伪造工作经历等不诚信行为
   - 对于此类请求，Agent SHALL 用温和但坚定的语气说明原因并引导回正题

**边界处理语气示例：**

| 用户说 | Agent 应回复 |
|---|---|
| "今天天气怎么样？" | "哈哈，天气确实会影响心情呢~不过话说回来，你最近求职季准备得怎么样了？有没有遇到什么困惑？" |
| "帮我写个Python爬虫" | "写代码不是我擅长的方向哦，不过如果你是想转行做技术开发，我可以帮你分析一下需要学什么、怎么规划~" |
| "你觉得XXX明星怎么样" | "哈哈我对娱乐圈不太了解呢~不过我们可以聊聊你的职业规划，比如你有没有特别想去的公司类型？" |
| "帮我编一段工作经历" | "简历最重要的是真实哦，虚假信息反而可能在背调时出问题。不如我们一起看看你现有的经历有哪些亮点可以更好地呈现？" |

5. THE Agent SHALL 在引导回求职话题时，优先选择**与用户背景或上次对话相关**的方向，而不是每次都用同一句通用引导语

### 需求 7：与用户简历数据打通

**用户故事：** 作为一名已上传简历的用户，我不希望 Agent 重复询问简历中已有的信息（如当前职业、技能、教育背景），而是直接基于已有数据开始深度对话。

#### 验收标准

1. THE 系统 SHALL 在 Agent 启动时自动检测用户是否已上传并解析简历
2. IF 用户已有已解析的简历数据，THE 系统 SHALL：
   - 将简历解析结果（parsed_data）注入到 Agent 的对话上下文中
   - 包括但不限于：候选人简介（candidate_summary）、技能、教育背景、工作经历、MBTI 类型
3. IF 用户尚未上传简历，THE Agent SHALL 正常运行（不报错），只是对话时不具备简历数据
4. THE Agent SHALL 在首次回复时（或通过 System Prompt 引导）自动知晓用户的背景信息，避免提出「你目前做什么工作？」「你学什么专业的？」之类已存在于简历中的问题
5. THE 当用户切换到「修改简历」模式时，THE Agent SHALL 自动读取已上传简历作为原始材料
6. THE 当用户切换到「换赛道」模式时，THE Agent SHALL 基于简历中的当前行业和技能进行分析
7. THE 当用户切换到「职业规划」模式时，THE Agent SHALL 基于简历中的经验年限和教育背景给出建议
8. THE 当用户切换到「面试模拟」模式时，THE Agent SHALL 基于简历中的技能和目标岗位生成面试题
9. THE 当用户上传新简历或切换已保存的简历时，THE Agent 上下文 SHALL 同步更新为当前简历的数据
10. THE 预设问题 SHALL 根据是否有简历数据做动态调整，具体见需求 11「初始态与预设问题」章节
11. THE 简历数据的注入 SHALL 为只读引用，Agent 的对话不会修改用户的简历数据

2. THE 后端 SHALL 扩展 `/chat` 接口以支持 `mode` 参数
3. THE 请求体 SHALL 包含：

```json
{
  "messages": [
    {"role": "user", "content": "我想换个赛道试试，帮我分析一下转型策略？"},
    {"role": "assistant", "content": "好的，让我先了解一下你的背景..."}
  ],
  "mode": "career_switch",
  "resume_context": {
    "has_resume": true,
    "resume_id": "resume_123",
    "candidate_summary": "2年前端开发经验，掌握React、Vue..."
  }
}
```

4. THE 响应体 SHALL 包含：

```json
{
  "reply": "从当前行业转到新方向是一个很好的规划...",
  "mode": "career_switch",
  "suggested_actions": ["学习目标行业基础知识", "参加相关培训"],
  "has_resume_template": false,
  "interview_state": null
}
```

5. THE 面试模拟模式 SHALL 有额外的状态字段 `interview_state`：
   - `question_index`: 当前问题序号
   - `total_questions`: 总问题数
   - `score`: 当前累计评分
   - `mode_type`: "technical" | "behavioral" | "comprehensive"

#### 智谱 AI GLM-4-Flash API 集成要求

6. THE 后端 SHALL 配置智谱 AI API 的以下参数（与项目现有配置一致）：
   - API Endpoint: `https://open.bigmodel.cn/api/paas/v4/`
   - Model: `glm-4-flash`
   - API Key: 从环境变量 `ZHIPU_API_KEY` 读取
   - Temperature: 0.7（可根据不同模式调整，面试模拟可设为 0.8-0.9 增加多样性）
   - Max Tokens: 900（限制输出长度，避免不必要的费用）
7. THE 系统 SHALL 使用 `openai` Python SDK 调用智谱 AI API（智谱兼容 OpenAI 的 API 格式，`base_url` 设为 `https://open.bigmodel.cn/api/paas/v4/`）
8. THE 系统 SHALL 处理智谱 API 的错误响应（如 rate limit、token 超限、API 不可用等）
9. THE 系统 SHOULD 实现请求重试机制（最多重试 3 次，指数退避策略）
10. THE 系统 SHOULD 记录 API 调用日志（请求时间、token 消耗、响应时间、模型版本）
11. THE 系统 SHALL 针对不同模式优化 API 调用参数：
    - 换赛道/职业规划：temperature=0.7，强调逻辑性和专业性
    - 修改简历：temperature=0.6，强调准确性和结构化
    - 面试模拟：temperature=0.8，增加问题多样性和真实感

### 需求 16：对话上下文长度

1. THE 对话历史 SHALL 在本地保存最近 50 条消息
2. THE 发送给 API 的消息列表 SHOULD 包含最近的 20 条消息（以控制 token 消耗）
3. THE 关键上下文信息（用户简历、职业背景）SHALL 持久化到 localStorage

### 需求 17：响应性能

1. THE API 响应时间 SHOULD < 5 秒（P95）
2. THE 打字指示器 SHALL 在等待响应时显示
3. THE 面试模拟模式 SHOULD 支持流式输出（SSE），模拟真实对话节奏

### 需求 18：向后兼容

1. THE 现有功能（简历上传、MBTI 分析、岗位推荐、简历诊断）SHALL 不受影响
2. THE 现有 `/upload` 接口 SHALL 继续正常工作
3. THE 新的 Agent 功能 SHOULD 可以访问现有的简历解析数据

### 需求 19：API 安全保护

**用户故事：** 作为开发者，我需要确保 API 密钥和其他敏感信息不会泄露到 GitHub 等公共代码仓库中，保护系统安全。

#### 验收标准

1. THE API 密钥（ZHIPU_API_KEY）SHALL 存储在环境变量中，**绝不**硬编码在源代码中
2. THE 项目 SHALL 包含 `.env.example` 文件，列出所有需要的环境变量，但不包含实际值
3. THE 实际的 `.env` 文件 SHALL 添加到 `.gitignore` 中，确保不会被提交到版本控制系统
4. THE 后端代码 SHALL 从环境变量读取 API 密钥：
   ```python
   import os
   from openai import OpenAI
   ZHIPU_API_KEY = os.getenv("ZHIPU_API_KEY")
   client = OpenAI(api_key=ZHIPU_API_KEY, base_url="https://open.bigmodel.cn/api/paas/v4/")
   ```
5. THE 部署文档 SHALL 明确说明如何设置环境变量
6. THE 项目 SHALL 包含安全检查脚本或配置，防止意外提交敏感信息
7. THE 代码审查流程 SHALL 包含检查是否包含硬编码的 API 密钥或其他敏感信息
8. THE 错误日志 SHALL 不包含完整的 API 密钥或其他敏感信息

#### 环境变量管理

1. **开发环境**：
   - 使用 `.env` 文件（已添加到 `.gitignore`）
   - 通过 `python-dotenv` 或类似库加载

2. **生产环境**：
   - 使用服务器环境变量
   - 使用 Docker 环境变量
   - 使用云服务提供的密钥管理服务（如 AWS Secrets Manager、Azure Key Vault）

3. **CI/CD 管道**：
   - 使用 CI/CD 平台的环境变量功能
   - 使用加密的 secrets

#### 防止意外提交的防护措施

1. **`.gitignore` 配置**：
   ```
   # 环境变量文件
   .env
   .env.local
   .env.*.local

   # 密钥文件
   *.key
   *.pem
   *.cert

   # 配置文件
   config.local.*
   secrets.*
   ```

2. **预提交钩子（pre-commit hook）**：
   - 检查是否包含硬编码的 API 密钥模式
   - 检查是否尝试提交 `.env` 文件
   - 检查是否包含常见的敏感信息模式

3. **代码扫描工具**：
   - 使用 `git-secrets` 或类似工具扫描代码库
   - 定期运行安全扫描

#### 应急响应计划

1. IF API 密钥意外泄露，THE 团队 SHALL：
   - 立即在智谱 AI 开放平台撤销泄露的密钥
   - 生成新的 API 密钥
   - 更新所有环境中的环境变量
   - 审查代码提交历史，找出泄露原因
   - 加强防护措施，防止再次发生
